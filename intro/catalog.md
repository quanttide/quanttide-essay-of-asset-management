# 从数据目录到资产目录：视角的转换与结构迁移札记

## 视角的转换：从客观数据到主观资产

在我们的治理框架中，区分数据与资产，并非简单的前后端界面划分，而是一次根本性的视角转换。同一条信息，在不同的业务场景下，因为我们看待它的视角不同，它所具备的意义就完全不同。

我们可以把数据工程、资产管理、开源管理看作是我们认知世界的不同视角。视角的不同，决定了我们关注什么，忽略什么。在数据工程的视角下，我们面对一串信息，可能会高度关注它是 JSON 还是 YAML，以及它们之间如何互相转换。但当我们切换到资产管理的视角时，底层的格式是 JSON 还是 YAML 已经不重要了，我们关注的是它所承载的业务逻辑和价值。

以一个 GitHub 仓库为例。如果我们是为了给某位老师做数据分析，将这个仓库的元数据存入数据云，那么它只是一条客观存在、冷冰冰的普通数据。但是，当我们把我们自己的 GitHub 仓库元数据接入到资产云时，它就具备了完全不一样的意义。如果此时我们的主观视角侧重于开源行为，那么它在我们的系统里就是一个开源仓库。资产，本质上就是被人类主观赋予了意义的数据。

## 核心思想：剥离结构与迁移对象

基于这种认知的转变，我们此次架构设计的核心建模思想并不是凭空创造一套新体系，而是执行一次精准的结构迁移。我们将数据工程中那套成熟的、严密的建模思想和关系结构提取出来，但是把被处理的对象从数据替换成了资产。我们是用看待数据的严谨结构体系，去承载和看待主观多变的资产。把这个核心问题讲清楚，是我们整个系统能够正确演进的前提，只有理解了这种视角的剥离与结构的迁移，AI 在后续的自动化建模和代码生成中，才知道该如何帮你权衡取舍。

## 架构落地：标准目录与动态契约的结合

为了让这种结构迁移能够落地，我们采用标准目录结合动态契约的模型来构建现实的资产台账。

标准目录代表了我们从数据工程中迁移过来的那套严密结构。它是一组绝对标准的基座字段，例如标识、名称、标题、描述以及所有者信息。无论我们的视角如何变化，这套承载资产的骨架是不变的。任何异构系统里的非结构化实体，在进入我们的视野时，都必须首先被装载进这个标准的结构容器中。

动态契约则负责承载那些随视角变化而产生的特定意义。因为现实总是比标准复杂，当我们以特定视角审视飞书或 GitHub 时，总有一些特定属性需要保留。我们在目录中预留了扩展空间，将这些具有特定视角价值的字段客观记录下来，交由契约文件在后续的生命周期中进行校验和治理。

## 映射法则：穿透表象与关注实际行为

在这种视角驱动的架构下，当我们把现实世界的资产注册进目录时，我们遵循的唯一法则就是关注实际行为，忽略表面名称。

当我们在资产管理的视角下映射一个 GitHub 仓库时，我们并不关心原系统的 API 叫什么。GitHub 社区经常习惯在 description 字段里写一句简短的说明。如果在数据工程的死板视角下，description 就必须映射给 description。但在我们资产视角的建模中，因为我们需要一个明确的业务名称，所以只要它表现为短句，我们就会穿透表象，将其直接映射到我们标准结构里的 title 字段中。

从数据到资产的跨越，就是一场摒弃技术执念、回归主观业务意图的视角重塑。建立资产目录，就是把这套融合了严谨结构与主观视角的映射法则固化下来，为整个系统的持续治理提供最坚实的现实台账。
